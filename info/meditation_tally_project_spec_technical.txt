Approved Product Behavior
-------------------------
The app behavior approved for version 1 is:

- The home screen shows today's date
- The main circular counter is the primary tap target
- Tapping the circle increments the tally by 1
- The current count is displayed in large type inside the circle
- A progress arc shows progress toward an optional target
- A small goal label such as "20 / 108" is shown near the main counter
- Undo reverses the most recent increment
- Reset requires confirmation
- History shows daily totals
- All data is stored locally on the device
- The app works fully offline

Important Design Decision for Undo
----------------------------------
Undo in version 1 should be session-scoped rather than full historical action-stack persistence.

Recommended behavior:
- Undo only reverses the most recent increment
- Undo is available while the app process is active or while the latest action is tracked in memory
- Reset clears undo availability
- App relaunch does not need to preserve a full undo stack for MVP

Alternative:
- Persist last action metadata if persistent undo after relaunch is required

Recommended MVP choice:
- In-memory last action tracking in ViewModel or repository-assisted state

Business Logic Rules
--------------------
1. Increment
- User taps the main circle
- Count increases by 1
- Record is saved immediately
- Progress is recalculated immediately
- Optional haptic feedback is triggered from the UI layer

2. Undo
- Only available if the last successful action was an increment
- Count decreases by 1
- Count cannot go below 0
- Updated value is saved immediately

3. Reset
- Reset action opens confirmation dialog
- On confirmation, today's count becomes 0
- Reset is saved immediately
- Previous daily history remains unchanged

4. Date Rollover
- On app launch, resume, or repository initialization, compare device current date with active app date
- If current date differs, ensure the new date has its own record
- Do not overwrite previous day values
- Start the new day at 0 unless a record already exists for that date

5. Goal Progress
- If target enabled and target > 0:
  progress = min(count / target.toFloat(), 1f)
- Numeric display may exceed target
- Progress arc visually caps at 100 percent

Date Handling Strategy
----------------------
Store dates in ISO-8601 string form:
- yyyy-MM-dd

Reasons:
- Easy sorting
- Easy Room primary key use
- Human-readable
- Stable for history grouping

Use local device date:
- LocalDate.now(clock)

Recommended testability:
- Inject Clock instead of calling LocalDate.now() directly in business logic

Home Screen Event Flow
----------------------
Tap counter:
1. User taps CounterTapCircle
2. UI optionally triggers haptic
3. UI sends TapCounter event to HomeViewModel
4. ViewModel calls increment use case / repository
5. Repository updates today's record in Room
6. DAO emits updated data
7. ViewModel updates HomeUiState
8. UI recomposes with new count and progress

Undo:
1. User taps Undo
2. ViewModel checks canUndo
3. Repository updates today's count to max(0, count - 1)
4. Undo availability is cleared
5. UI recomposes

Reset:
1. User taps Reset
2. ViewModel sets showResetDialog = true
3. User confirms
4. Repository sets today's count = 0
5. ViewModel hides dialog and clears undo state
6. UI recomposes

History Loading Flow
--------------------
1. HistoryViewModel subscribes to observeAllRecords()
2. DAO emits records ordered by newest date first
3. ViewModel maps entities to UI models
4. UI renders empty state or list

Error Handling Strategy
-----------------------
MVP should keep errors simple and safe.

Expected error scenarios:
- Database write failure
- Corrupt or invalid preference value
- Unexpected date parsing issue

Recommended behavior:
- Fail gracefully
- Keep last known UI state where possible
- Log recoverable failures
- Show generic non-intrusive feedback only if needed

Because this app is local-first and simple, most errors should be rare.

State Synchronization
---------------------
Single source of truth:
- Room is the source of truth for daily records
- DataStore is the source of truth for preferences

Do not keep an unsynced duplicate count state in multiple places.

Recommended approach:
- Observe today's record from Room
- Observe preferences from DataStore
- Combine them in ViewModel into one HomeUiState
